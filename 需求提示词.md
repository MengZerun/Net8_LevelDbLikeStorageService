# 需求提示词：使用 C# .NET 8.0 仿实现 LevelDB 数据库服务软件

请基于以下需求，使用 C# .NET 8.0 实现一个尽可能仿照现有 `storage_0729` 数据库服务软件的数据库服务项目。项目需要保存为一个完整、可编译、可运行、可发布的 .NET 8.0 解决方案。日志系统必须使用 `log4net`。目标是在行为、配置文件、HTTP 接口、请求 JSON 格式、返回 JSON 格式、目录结构和运维方式上尽量兼容现有数据库服务。

## 一、背景说明

现有数据库服务软件位于：

```text
C:\Users\z\Desktop\storage_0729\storage_0729\bin
```

现有服务是一个封装 LevelDB 的数据库服务软件。C# 业务侧通过 `D:\CSharpLibs-20260603\Common\ComOther.cs` 中的 `GetLevelDbOpCmd` 方法生成数据库操作 JSON 命令，然后通过 HTTP POST 到数据库服务端口，完成数据库的增删改查、前缀查询、批量写入等操作。

现有服务的配置文件目录为：

```text
C:\Users\z\Desktop\storage_0729\storage_0729\bin\config
```

核心配置包括：

```text
config.json
db.json
db-cs.json
db_config.json
server_config.json
log_config.json
get_all_file.json
```

现有服务默认 HTTP 端口为：

```text
9877
```

现有 C# 业务请求中常见目标地址为：

```text
http://192.168.8.2:9877
http://127.0.0.1:9877
```

新项目需要在 .NET 8.0 下实现一个功能类似、协议尽量兼容的数据库服务。

## 二、总体目标

实现一个名为 `Net8_LevelDbLikeStorageService` 的 C# .NET 8.0 数据库服务，尽可能复刻现有数据库服务的外部行为。

必须满足：

1. 使用 C#。
2. 使用 .NET 8.0。
3. 使用 ASP.NET Core Minimal API 或 ASP.NET Core Web API 提供 HTTP 服务。
4. 日志系统使用 `log4net`。
5. 支持 Windows 控制台运行。
6. 支持作为 Windows Service 运行。
7. 支持读取 `config` 文件夹中的 JSON 配置。
8. 支持逻辑库名 `db_name` 到物理数据库目录的映射。
9. 支持和现有 `GetLevelDbOpCmd` 生成命令兼容的 JSON 请求。
10. 支持基础数据库操作：连接/初始化、断开/释放、增、删、改、查。
11. 支持 `put`、`get`、`delete`、`list` 操作。
12. 支持 `op_mode`：`all_ow`、`all`、`last`、`prefix_keys`、`prefix_kvs`、`ap`、`kv`。
13. 支持 `is_batch` 批量写入。
14. 返回 JSON 格式尽可能兼容现有服务，至少包含 `resultCode`、`msg`、`value`、`uniqueKey`。
15. 项目中提供详细 README、示例配置、示例请求、C# 客户端示例。
16. 提供单元测试或集成测试，覆盖基本 CRUD。

## 三、重要兼容协议

### 3.1 请求方式

服务端需要监听 HTTP POST 请求。

默认地址：

```text
http://0.0.0.0:9877
```

客户端请求：

```http
POST / HTTP/1.1
Content-Type: application/json
```

Body 是一个 JSON 对象。

也需要兼容以下路径：

```text
POST /
POST /db
POST /api/db
POST /api/storage
```

这几个路径都进入同一个数据库命令处理逻辑。

### 3.2 标准请求 JSON

必须兼容现有 `GetLevelDbOpCmd` 输出的请求格式：

```json
{
  "db_name": "tray",
  "is_batch": "false",
  "key": "T20260603001-001",
  "key_list": "",
  "op_mode": "all_ow",
  "operation": "put",
  "uniqueKey": "20260603120000000000insertPcsToTrayDb00000001",
  "value": "{\"trayCode\":\"T20260603001\",\"pos\":1,\"finalRes\":\"OK\"}"
}
```

字段要求：

| 字段 | 类型 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| `db_name` | string | 是 | 逻辑库名 |
| `operation` | string | 是 | 操作类型：`put`、`get`、`delete`、`list` |
| `op_mode` | string | 是 | 操作模式 |
| `key` | string | 是 | key |
| `value` | string/object/array | 是 | 写入值或空字符串 |
| `uniqueKey` | string | 是 | 请求唯一标识 |
| `key_list` | string/array | 是 | key 列表，不用时为空字符串 |
| `is_batch` | string/bool | 否 | 是否批量 |

必须同时兼容 `is_batch` 为字符串和布尔值：

```json
"is_batch": "false"
```

```json
"is_batch": false
```

如果 `value` 是对象或数组，服务端应自动压缩序列化为 JSON 字符串后保存。

### 3.3 兼容旧字段命名

现有 `ComOther.GetLevelDbOpCmd` 另一个重载会生成以下字段：

```json
{
  "key": "k1",
  "op": "put",
  "dbName": "tray",
  "opMode": "all_ow",
  "value": {},
  "uniqueKey": "xxx",
  "more": "sendMore"
}
```

新服务需要尽量兼容该格式，将字段映射为：

```text
op      -> operation
dbName  -> db_name
opMode  -> op_mode
```

### 3.4 标准返回 JSON

所有请求返回 JSON。

成功：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "",
  "uniqueKey": "请求中的 uniqueKey"
}
```

失败：

```json
{
  "resultCode": -1,
  "msg": "错误原因",
  "value": "",
  "uniqueKey": "请求中的 uniqueKey"
}
```

查询成功：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "{\"trayCode\":\"T20260603001\",\"pos\":1,\"finalRes\":\"OK\"}",
  "uniqueKey": "请求中的 uniqueKey"
}
```

前缀查询成功：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "[{\"key\":\"T20260603001-001\",\"value\":\"{\\\"finalRes\\\":\\\"OK\\\"}\"}]",
  "uniqueKey": "请求中的 uniqueKey"
}
```

注意：为了兼容现有 C# 业务代码，`value` 应返回字符串，而不是直接返回对象或数组。也就是说，复杂值需要二次序列化成字符串。

## 四、存储引擎要求

### 4.1 优先方案

优先使用 .NET 8.0 下可稳定运行的 LevelDB 兼容库。

如果可以稳定使用 LevelDB 原生绑定，则实现真正 LevelDB 存储。

可评估：

```text
LevelDB.Standard
leveldb-sharp
LevelDB.NET
```

但需要注意 Windows x64 native dependency 是否可靠。

### 4.2 允许的工程化折中

如果 .NET 8.0 下 LevelDB 绑定不可稳定使用，允许在内部实现一个可替换的存储抽象：

```csharp
public interface IKvDatabase
{
    Task PutAsync(string key, string value, CancellationToken ct);
    Task<string?> GetAsync(string key, CancellationToken ct);
    Task<bool> DeleteAsync(string key, CancellationToken ct);
    Task<IReadOnlyList<string>> ListKeysByPrefixAsync(string prefix, CancellationToken ct);
    Task<IReadOnlyList<KeyValuePair<string, string>>> ListKvsByPrefixAsync(string prefix, CancellationToken ct);
    Task CloseAsync();
}
```

然后提供至少一个真实持久化实现。

可接受的备选实现：

1. RocksDbSharp：作为 LevelDB 风格 KV 存储，磁盘持久化，性能接近。
2. LiteDB：仅作为开发 fallback，不作为默认生产引擎。
3. 自研文件 KV：仅作为测试 fallback，不作为默认生产引擎。

默认生产建议：

```text
RocksDbSharp 或可用的 LevelDB binding
```

无论底层使用 LevelDB 还是 RocksDB，对外协议必须保持 LevelDB 服务风格，不暴露底层差异。

### 4.3 多库管理

服务启动时读取 `config/db.json`。

每个逻辑库配置结构：

```json
{
  "tray": {
    "path": "D:/dlib/tray",
    "status": "active",
    "version": "0.0.0.1"
  }
}
```

要求：

1. `status != active` 的库不打开。
2. 请求未配置的 `db_name` 时返回错误。
3. 请求 inactive 的 `db_name` 时返回错误。
4. 物理目录不存在时自动创建。
5. 每个 `db_name` 对应一个数据库实例。
6. 服务退出时关闭所有数据库实例。
7. 同一个库需要线程安全。
8. 多客户端并发访问时不能损坏数据。

## 六、数据库操作详细行为

### 6.1 put + all_ow

请求：

```json
{
  "db_name": "tray",
  "is_batch": "false",
  "key": "k1",
  "key_list": "",
  "op_mode": "all_ow",
  "operation": "put",
  "uniqueKey": "u1",
  "value": "{\"a\":1}"
}
```

行为：

1. 打开/获取 `tray` 数据库实例。
2. 将 key `k1` 写入。
3. 如果 key 已存在，覆盖原值。
4. 返回 `resultCode = 0`。

返回：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "",
  "uniqueKey": "u1"
}
```

### 6.2 put + ap

请求：

```json
{
  "db_name": "tray_list",
  "key": "trayCodeList",
  "operation": "put",
  "op_mode": "ap",
  "value": "T001",
  "uniqueKey": "u1",
  "key_list": "",
  "is_batch": "false"
}
```

行为：

1. 读取原值。
2. 如果原值为空，创建 JSON 数组。
3. 如果原值是 JSON 数组字符串，把当前 `value` 追加进去。
4. 如果原值不是 JSON 数组，可兼容为字符串列表或返回错误。
5. 写回。

建议保存格式：

```json
["T001","T002"]
```

注意：这是根据现有 `op_mode = ap` 推断出的兼容语义。必须在 README 中说明。

### 6.3 put + kv + is_batch=true

请求：

```json
{
  "db_name": "tray",
  "is_batch": "true",
  "key": "",
  "key_list": "",
  "op_mode": "kv",
  "operation": "put",
  "uniqueKey": "u1",
  "value": [
    {
      "key": "k1",
      "value": "{\"a\":1}"
    },
    {
      "key": "k2",
      "value": "{\"a\":2}"
    }
  ]
}
```

行为：

1. 解析 `value` 为数组。
2. 每个元素必须包含 `key` 和 `value`。
3. 批量写入。
4. 尽可能使用底层 WriteBatch。
5. 如果任意一条失败，返回错误。

兼容要求：

`value` 可能已经被现有 `GetLevelDbOpCmd` 序列化成字符串：

```json
"value": "[{\"key\":\"k1\",\"value\":\"{\\\"a\\\":1}\"}]"
```

服务端必须支持这两种形式。

### 6.4 get + all

请求：

```json
{
  "db_name": "tray",
  "key": "k1",
  "operation": "get",
  "op_mode": "all",
  "value": "",
  "uniqueKey": "u1",
  "key_list": "",
  "is_batch": "false"
}
```

行为：

1. 查询完整 key。
2. 找到则返回 value 字符串。
3. 找不到时 `resultCode` 建议仍为 `0`，`value` 为空字符串，`msg` 为 `not found`；或者 `resultCode = 404`。为了兼容现有业务，推荐 `resultCode = 0` 且 `value = ""`。

返回：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "{\"a\":1}",
  "uniqueKey": "u1"
}
```

### 6.5 get + last

行为要求：

1. 如果存在完整 key，优先按完整 key 查询。
2. 如果不存在，允许按 key 前缀查询并返回排序后最后一条。
3. 排序规则默认使用 ordinal 字符串升序。
4. README 中明确说明 `last` 的兼容语义。

这是为了兼容现有业务中：

```text
getTrayCodeByDmCode -> op_mode: last
getTrayInfoDataByTrayCode -> op_mode: last
```

### 6.6 delete + all

请求：

```json
{
  "db_name": "tray",
  "key": "k1",
  "operation": "delete",
  "op_mode": "all",
  "value": "",
  "uniqueKey": "u1",
  "key_list": "",
  "is_batch": "false"
}
```

行为：

1. 删除完整 key。
2. key 不存在时也返回成功，保持幂等。

### 6.7 list + prefix_keys

请求：

```json
{
  "db_name": "tray",
  "key": "T001",
  "operation": "list",
  "op_mode": "prefix_keys",
  "value": "",
  "uniqueKey": "u1",
  "key_list": "",
  "is_batch": "false"
}
```

行为：

1. 遍历数据库。
2. 找出所有以 `key` 为前缀的 key。
3. 按 key 升序返回。
4. `value` 字段返回 JSON 数组字符串。

返回：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "[\"T001-001\",\"T001-002\"]",
  "uniqueKey": "u1"
}
```

### 6.8 list + prefix_kvs

行为：

1. 遍历数据库。
2. 找出所有以 `key` 为前缀的 KV。
3. 按 key 升序返回。
4. `value` 字段返回 JSON 数组字符串。

返回：

```json
{
  "resultCode": 0,
  "msg": "success",
  "value": "[{\"key\":\"T001-001\",\"value\":\"{\\\"a\\\":1}\"},{\"key\":\"T001-002\",\"value\":\"{\\\"a\\\":2}\"}]",
  "uniqueKey": "u1"
}
```

## 十三、并发与性能要求

1. 支持多客户端并发 HTTP 请求。
2. 对同一个 `db_name` 的写入需要线程安全。
3. `put/delete/get` 不应全局锁住所有库。
4. 可以按库加锁，或使用底层引擎自身线程安全能力。
5. 前缀查询可能遍历大量数据，需要避免一次性占用过高内存。
6. 对 `list` 操作增加可选限制：

```json
{
  "limit": 10000
}
```

但为了兼容旧协议，默认不要求客户端传 `limit`。

7. 大请求体需要限制，默认建议 100 MB，可配置。

## 十四、磁盘检查要求

读取 `diskCheckSpace`：

```json
"diskCheckSpace": 1.0
```

写操作前检查目标数据库所在磁盘剩余空间。

如果剩余空间小于阈值，拒绝写入：

```json
{
  "resultCode": -601,
  "msg": "disk free space is lower than threshold",
  "value": "",
  "uniqueKey": "u1"
}
```

读操作不受影响。

## 二十二、关键实现建议

建议核心类如下：

```csharp
public sealed class DbCommand
{
    public string DbName { get; set; } = "";
    public string Operation { get; set; } = "";
    public string OpMode { get; set; } = "";
    public string Key { get; set; } = "";
    public string Value { get; set; } = "";
    public string UniqueKey { get; set; } = "";
    public string KeyList { get; set; } = "";
    public bool IsBatch { get; set; }
}
```

```csharp
public sealed class DbResponse
{
    public int ResultCode { get; set; }
    public string Msg { get; set; } = "";
    public string Value { get; set; } = "";
    public string UniqueKey { get; set; } = "";
}
```

```csharp
public interface IDbCommandHandler
{
    Task<DbResponse> HandleAsync(DbCommand command, CancellationToken ct);
}
```

```csharp
public interface IDatabaseManager
{
    IKvDatabase GetDatabase(string dbName);
    Task InitializeAsync(CancellationToken ct);
    Task CloseAllAsync();
}
```

所有 HTTP 请求进入后：

```text
Raw JSON
  -> DbCommandParser
  -> DbCommandValidator
  -> DbCommandHandler
  -> DatabaseManager
  -> IKvDatabase
  -> DbResponse
```

## 二十三、特别注意

1. 不要设计成 SQL 数据库。
2. 不要要求客户端建立 TCP 长连接。
3. 不要改变现有 JSON 字段名。
4. 不要把返回 `value` 改成对象；为了兼容应保持字符串。
5. 不要让一个坏请求导致服务退出。
6. 不要在写入时把对象格式化成带缩进 JSON，应使用紧凑 JSON。
7. 不要默认启用危险管理接口。
8. 不要把日志写死到系统盘，应读取配置。
9. 不要把 `D:/dlib` 写死在代码里，应来自配置。
10. 不要假设所有客户端都是 .NET 8；必须兼容 .NET Framework 4.6.2 生成的请求。
